A

语料完整性:强度不足,过度依赖随机性

引用完整性本是可机器校验、可机器修复的问题,当前却在打分前不校验、修复又交给随机 agent。这是今天那个 bug 的真正温床。

01 打分前缺少 CI 绿灯门禁 高危可离线
证据src/governed_bi/eval/run_experiment.py — 全程未调用 validate_corpus
现象run_experiment 构建 baseline / curated / curated_sme 三臂语料后直接打分,从不在打分前对语料跑一次校验。这正是那 12 条悬空 term 绑定能无声混入被打分 A3 语料的通道。
影响带引用完整性缺陷的语料会照常产出 EX 数字,且没有任何信号——headline 数字可能被语料 bug 污染而无人察觉。
建议打分前逐臂调用 validate_corpus,把 validate_finding_count 写进 summary.json;非绿灯时 fail-closed 或显著告警。用 beer_factory 的 SQLite 夹具即可离线单测。
02 确定性引用修复只覆盖 term 绑定 中可离线
证据asset_bag.repair_term_bindings()(今日新增,仅处理 term.binding)
现象其余可确定性修复的悬空引用——column.references、metric.base_table、join 两端、rule.scope——仍旧交给随机的 agent fix-pass 去猜。
影响只有 term 一类被兜住;其余类型一旦悬空,又会落回那条会崩溃、会造重复的随机修复路径。
建议抽象出通用 repair_references(),把所有可解析的引用类型都在代码里确定性修复,让整个 fix-pass 摆脱随机性。
03 join 的 on 子句在离线阶段从不校验 中偏高可离线
证据corpus/validate.py — on 列存在性注释为"deferred to the eval harness / 需活库"
现象validate 只校验 join 两端的 id,不校验 on 子句引用的列是否属于这两张表;on 是一段原始 SQL,离线从不解析。
影响一个 on 列写错或幻觉的 join 会离线 CI 全绿,只在服务期暴露——或更糟,静默错连。
建议sqlglot 已是依赖,用它解析 on,校验列归属于 join 的两张表。无需活库。
04 写工具按 slug(name) 创建,无法按 id 更新 中可离线 · 改动较大
证据asset_bag.upsert_term / upsert_metric … — tid = f"term_{schema}_{slug(name)}"
现象upsert_* 用 slug(name) 当键;agent 被要求"修复资产 X"时会新建一个,而非编辑 X。
影响这正是 6 → 12 翻倍的机制。工具接口缺少按 id 寻址的更新语义。
建议增加按 asset_id 寻址的更新路径;或让 upsert 在传入既有 id 时执行原地更新。
B

静默失败 / fail-open

失败被降级为字符串、run 照常继续。单独看每处都合理,叠加缺口 #1 后就成了"坏语料被静默打分"的完整链条。

05 _invoke_agent 吞掉 agent 崩溃 高危可离线
证据curator/pipeline.py:253 — except Exception → 记为字符串,run 继续
现象agent.invoke 的任何异常被捕获、记进 fix_pass_error,run 照常往下走。与缺口 #1 叠加,fix-pass 崩溃或空跑都不会阻断流程。
影响失败不可见——一个静默空跑的 fix-pass 会留下坏语料被打分。
建议与 #1 的绿灯门禁配合;崩溃 / 空跑要么 fail-closed,要么升级为显式的 run 级信号,而非只写进一个易被忽略的字段。
06 fold 与 fix-pass 复用同一 agent 实例 中可离线
证据pipeline.py:549 / 720 — _validate_fix_pass(agent if agent_ran …)
现象Phase B 的 fold 与随后的 fix-pass 复用同一个已编译 agent,跨两次 invoke 的共享状态脆弱;最初那个 KeyError 就与这种复用相关。
影响难以复现的随机崩溃;状态耦合让 fix-pass 的行为依赖 fold 留下的痕迹。
建议每次 invoke 重建 agent,或显式隔离两次调用的状态。
C

对抗式校验只是桩

文档里的"独立 LLM 对手"是设计的一道防线,代码里还是空的。

07 refute() 是 NotImplementedError 中需测试环境
证据curator/adversary.py:104 — refute() raise NotImplementedError
现象设计里的"独立 LLM 对手 / refute-first"未落地,管线只跑结构化的 review() 信号。
影响文档化的对抗式校验与实现之间存在缺口;curated 资产缺了"独立反驳"这道防线。
建议落地 refute-first agent(需活模型);在此之前,文档应明确 review() 是当前唯一在跑的信号。
D

评测台的工程缺口

这些多与规模跑的可信度相关,部分已在文档追踪(标记 ◇ 已追踪),但仍是工程上未闭合的项。

08 grader 自检仅在 5 行样本上验证 中需环境
证据run_experiment.py:264 — sample=min(5, len(test))
现象gold-hash 自检只在 5 行样本上与参考实现比对,不是全量 head-to-head。
影响normalizer 漂移 / 边角用例可能逃过自检。
建议扩大抽样,或做一次全量对齐运行作为基线。
09 multi_schema=True 端到端从未在评测台跑通 高危布线可离线 · 验证需环境◇ 已追踪
证据run_experiment.py:240-246 硬编码 multi_schema=False;docs/plans/schema-qualification-scale-risk.md
现象69-schema 规模跑属于 multi_schema=True,而评测台从未在该模式下端到端跑过;mode-flag 从 DataSourceConfig → guardrail → context 的线路未验证(guardrail 单元逻辑已测,评测台布线未测)。
影响任一环节漏接会表现为静默的 L4 拒绝,拉低 EX,却被误读为模型 / 语料弱点——恰好污染规模跑的 headline。
建议给 summary.json 加"因限定不匹配而拒绝"的计数器;先跑 2-3 schema 预检,再上全量 69。
10 拒答门禁的误拒率无法度量 中需构造负样本◇ 已追踪
证据eval-ladder-results.md · 已知限制(BIRD 每题皆可答)
现象所有 BIRD 题都可答,评测从不触发拒答,误拒率(false-refusal)无从测量。
影响拒答的召回可测、精度不可测——一半的门禁质量是盲区。
建议构造 held-out 不可答集(跨库 + 移除覆盖 + 手写域外),度量拒答准确率与误拒率。
11 全流程单一随机种子,无方差 高危需环境◇ 已追踪
证据eval-ladder-results.md · Threats to validity(single seed)
现象所有臂都只有单一种子;curated 虽是确定性代码,其 deep-agent 探索却 run-to-run 会变。
影响任何 delta 只是方向性、不可下结论;v5 里"SME 首次抬升 EX"尤其需要复现才能站住。
建议≥3 种子跑,报告 mean±spread,再引用任何数字。
E

卫生 / 小项

不紧急,但会持续制造噪音或掩盖真实 bug。

12 68 处宽泛 except 吞异常 低偏中可离线
证据src/ 下 68 处 except Exception / 裸 except
现象部分是合理的(如 run_probe_query 把异常转成给模型的 error: 字符串),但这种模式整体上会掩盖 bug。
影响真实 bug 被降级为字符串,难以定位。
建议审计 curator / eval 路径上的宽泛捕获,收窄异常类型,或至少记录结构化上下文。
13 已删除模块的残留字节码 低可离线
证据__pycache__/gold.cpython-313.pyc(gold.py 已删)
现象已退休模块留下的 .pyc 残留。
影响纯卫生问题;可能误导 grep / 导入。
建议清理 __pycache__;确认 .gitignore 已覆盖。
